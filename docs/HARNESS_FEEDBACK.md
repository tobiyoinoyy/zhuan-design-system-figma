# Harness 反馈说明(给团队伙伴看)

> 你用 code agent(Cursor / Codex / Claude Code 等)生成转转 C 端业务页面,生成结果不满意时——这份文档告诉你怎么提反馈、提交后会发生什么。

---

## 一、什么时候提反馈

只要 agent 生成出来的页面让你觉得不对劲,就可以提。具体场景:

- **组件用错了**:它造了个 custom div 模拟卡片,而不是用现成的 `<ProductCardH>`
- **字段塞错东西**:`<Price promo>` 写成了一长段促销说明,价格右下小字溢出
- **低保真被照搬**:你给的低保真上随手画了个灰底,agent 把灰底当真做出来
- **营销跑偏 / 工具页加营销味**:工具型页面里出现 banner、轮播、渐变背景
- **信息层级不对**:核心动作藏在第三屏、关键信息字号还没旁边的小字大
- **怪、看不出是设计系统**:整体风格和其他页面不像一家人

不用纠结归类——填表时选「不确定」也行,维护者会复核。

---

## 二、怎么提

### 入口

主仓 https://github.com/tobiyoinoyy/zhuan-design-system-figma → Issues → New issue → 选 **Harness 反馈(生成结果不满意)** 模板。

模板还没建好之前,也可以直接发普通 issue,在标题前缀加 `[Harness Feedback]`。

### 要带什么材料

模板里都列出来了,核心是这几样:

1. **原始指令**——你当时给 agent 说的话(完整 prompt 或对话片段)
2. **生成结果**——截图、本地预览路径、关键代码片段,任选
3. **不满意点**——一句话说哪里不对
4. **期望效果**——你期望的样子,文字描述就行
5. **场景**——买/卖/租/营销/其他
6. **怀疑问题类型**——选最像的,不确定就选「其他」
7. **反馈类型**——A 客观规则违反 / B 专业判断跑偏 / C 个人审美偏好

### 关于「反馈类型」怎么选

- **A 客观规则违反**:有明确对错。组件用错了、字段超长了、用了不在 tokens 里的颜色——这些都是 A。
- **B 专业判断跑偏**:涉及"是不是合理"。信息层级、营销视觉边界、3 秒能不能识别——这些是 B。
- **C 个人审美偏好**:"我觉得这颜色不好看"、"我喜欢另一种字体"——这些是 C。**C 类不会进 Harness**,模板里也写了,这是为了避免规则库被审美偏好堆肥(每个人审美不同,沉淀进去会打架)。

不确定 A 还是 B 就选 B,维护者会复核。

## 三、提交后会发生什么

### 处理节奏

维护者每周或两周裁判一批反馈。**不是每条反馈都进 Harness**,只沉淀三类:

1. 高频问题(多次出现)
2. 会误导 agent 的语义边界
3. 一旦出错明显影响交付质量

单次审美偏好、一次性业务特例、需求本身没写清——不进 Harness,会被关 issue。

### 你会看到的状态变化

Issue 上会被打 label,代表维护者的判断:

| Label | 含义 |
|-|-|
| `harness-feedback` | 你提交时自动带上,代表"是一条反馈" |
| `source-confirmed` | 真实发生证据清楚,可以进入 Harness 沉淀 |
| `source-suspected` | 看起来像但证据不够,先观察累计 3 例再说 |
| `source-example` | 单次示例,不当高频问题处理 |
| `source-rejected` | 不属于 Harness 范围(需求问题/审美/特例),会关闭 |
| `accepted-to-harness` | 已经写进规则文档,等下次升级生效 |
| `needs-more-info` | 信息不够,需要你补充 |

### 升级到生效要多久

- confirmed → 进 docs / 自查清单 → 打 tag(如 `v-harness-2026-MM-DD`)→ 同步到使用方
- 一般两周内能等到一次升级
- 升级时会附一份「这次解决了什么」非技术说明,你能看明白哪些反馈被吸收了

---

## 四、你不需要做的事

- ❌ 不需要写规则、不需要改 docs/、不需要直接提 PR
- ❌ 不需要判断"这是不是高频问题"——你只描述"这次不满意",频率由维护者根据多条反馈统计
- ❌ 不需要给修复方案——你说期望是什么就行,怎么实现是 Harness 体系的事
- ❌ 不需要纠结归类——选错了维护者会改

**职责边界:伙伴只负责"不满意的素材",维护者负责"裁判和沉淀"**。这不是不信任你的判断,而是为了让规则有合法性——同一类问题被多个伙伴独立反馈,才能确认是真高频,而不是某次个人偏好。

---

## 五、如果你的反馈被 reject 了

不会被指责。被 reject 通常是因为:

- 这次的问题确实是需求本身没写清,补一句话就好(下次不会再发生)
- 是个人审美偏好(不进 Harness,但你可以在生成时直接和 agent 说"换个颜色")
- 是一次性业务特例(为它写规则反而会绑死其他场景)

reject 的 issue 维护者会关闭并写明原因。如果你不同意,可以在 issue 里继续讨论——给出新证据(比如又遇到一次)就可以重开。
