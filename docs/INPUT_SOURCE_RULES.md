# 输入源分流规则 (INPUT_SOURCE_RULES) — V0.3

> 这份文档是 **流程层 Harness**:告诉 code agent 拿到伙伴给的材料后,第一步先判断输入源类型,再按对应路径开工——不要不分类型直接发挥。
>
> 配套:产物层规则在 [`COMPONENT_USAGE.md`](./COMPONENT_USAGE.md)、设计硬约束在 [`DESIGN.md`](./DESIGN.md)、生成流程顺序在 [`../AGENTS.md`](../AGENTS.md) §2。本文只写 **「判断输入源 + 选路径」** 这一层,不重写它们。
>
> Agent 看到本文档某一段,意味着:你接下来生成代码,必须按这一段的处理路径走,否则会被反馈回流抓出来沉淀新规则(见 [`HARNESS_FEEDBACK.md`](./HARNESS_FEEDBACK.md))。

## 重要前提:这是 Harness 规则,不是组件运行时校验

> 本文写的「禁止 X」「只允许 Y」「先判断后开工」等约束,**不是组件源码的运行时校验**——
> 即使 agent 不读这份文档直接发挥,组件**照样会渲染出来**,只是流程跑偏(错的输入路径生出错的页面)。
>
> 这份文档的执行机制:
>
> 1. **agent 自觉**:开工前先判断输入源,按本文路径走
> 2. **AGENTS.md 入口接线**:让 agent 进仓库就被引导读本文(详见 [`../AGENTS.md`](../AGENTS.md))
> 3. **反馈回流**:伙伴跑出问题反馈给主仓维护者,沉淀到 [`CHANGELOG-RULES.md`](./CHANGELOG-RULES.md)、[`HARNESS_ISSUES.md`](./HARNESS_ISSUES.md)
>
> 自动 audit 扫不到流程层(流程不留可扫描的代码痕迹),所以这一层比产物层更依赖 agent 自觉 + AGENTS.md 引导 + agent 自查清单(见 [`COMPONENT_USAGE.md`](./COMPONENT_USAGE.md) §"生成自查清单")。

## 0 · 第一步:判断输入源类型

伙伴给你的可能是以下任意一种或几种混合。**开工前先判断,再选路径**:

| 输入源 | 特征 | 对应章节 |
|-|-|-|
| 文字指令(清晰) | 一句话讲清"做什么页、给谁用、要哪些信息" | §1 |
| 文字指令(模糊) | 只有目标没有细节,或细节互相打架 | §1 |
| 低保真原型 | 灰白线框、纯结构示意,颜色/字号不准 | §2 |
| 高保真原型 | 已有视觉、颜色/字号/间距都看着像真的 | §3 |
| PRD / 业务上下文 | 文字+表格的产品需求文档 | §4 |
| 营销 / 大促 / 节日活动 | 关键词命中 [`DESIGN.md`](./DESIGN.md) §8.1 | §5 |
| 混合输入 | 上面任意几种叠加 | §6 |

文字类输入判断不出来时:默认按"文字指令(模糊)"走 §1,先反问一句澄清,不要硬开工。

### 0.1 图片附件特别说明(HARNESS-INPUT-V0.1)

**图片附件不是默认视觉稿。** 伙伴上传图片时,可能没说是低保真还是高保真,或者说了但和图片实际特征对不上。开工前 agent 必须先自己判断。

**判断依据以图片特征为准,不完全依赖伙伴说法**——伙伴可能把 Figma 灰框稿叫"高保真",也可能反过来。

| 角色 | 特征 | 走哪节 |
|-|-|-|
| 低保真 | 灰框线框、手绘、黑白为主;商品图是占位;字体/间距/颜色随意;模块名像"banner/按钮/列表/占位图" | §2 |
| 高保真 | 有明确颜色/字体/间距/真实图标;接近最终视觉稿;模块层级和状态表达完整 | §3 |
| 判断不出来 | 介于两者之间,或既有线框又有局部高保真模块 | **按低保真处理(§2)**,避免污染设计系统 |

**伙伴说法和图片特征冲突时**:

- 优先按图片特征判断
- 仍不确定的,按低保真处理
- 在回复里说一句"你说是 X,但图片更像 Y,我按 Y 的规则处理",让伙伴看到判断过程

**多张图**:每张图分别判断角色,不要全部当视觉稿。混合附件的处理路径见 §6。

---

### 0.2 复杂图片输入的执行节奏(HARNESS-INPUT-V0.2)

**为什么有这条**:图角色判清了,不等于可以一口气从图直接生成完整页面。复杂附件(多屏 / 多状态 / 流程跳转)一次性生成容易卡在"理解完了但没落文件"——伙伴等了半天看不到产物。这条规则强制 agent 把「先理解,再生成」拆成两段。

**触发条件(任一命中即触发)**:

- ≥3 张图,或单图明显是多屏拼接(≥2 屏纵向/横向拼接)
- 图上有流程箭头、状态转移线、跳转关系
- 同一页面有 ≥3 个状态(默认/已选/禁用/加载/错误等)
- 伙伴明说"以原型为准 / 严格还原 / 不能删减信息"
- 单次输入 ≥4 个独立模块/页面

**触发后的执行节奏**:

1. **第一段:先输出「附件理解摘要」,不直接进入代码生成**。摘要内容:
   - 图角色判断(每张图按 §0.1 标低/高/判不出)
   - 识别出的页面 / 模块 / 流程清单(一句话一项,不展开)
   - 必须保留的信息和交互(伙伴明说不能删的、原型上明显的核心交互)
   - 视觉样式按 §2 / §3 处理(不在摘要里展开禁止/继承清单,引用即可)
   - 大致组件映射(开工前一句话说会用哪些 `src/components/*` 里的真组件、哪些区域是 custom module,按 §3 写法)
2. **第二段:等伙伴回复**。三种走向:
   - 伙伴说"按这版做" / "OK" / 任何明确确认 → 直接进入生成,开工前补一句"按摘要执行"
   - 伙伴有修订意见 → 改摘要,再等下一轮确认
   - **伙伴一个回合内没回复** → 默认按摘要继续生成,开工前必须主动说一句"未收到异议,按摘要执行;如有偏差请在生成后提反馈(按 [`HARNESS_FEEDBACK.md`](./HARNESS_FEEDBACK.md))"

**为什么要这么严**:摘要这一步不是"问伙伴怎么做",是 agent 自己把要做什么先写明白避免走偏。伙伴不回复也能继续,不会变成新的卡点。但留追溯——出问题时能从摘要看到 agent 当时怎么理解的。

**不触发的情况**:单张图 + 单页面 + 状态 ≤2 + 伙伴没要求严格还原——直接按 §2 / §3 处理就行,不强制摘要。

## 1 · 文字指令(清晰 / 模糊)

### 1.1 清晰文字指令

特征:包含「做什么页 + 给谁用 + 要哪些核心信息 + 是否营销场景」。

**处理路径**:直接进 [`../AGENTS.md`](../AGENTS.md) §2 标准流程(① 组件 → ② starter → ③ 泛化)。

### 1.2 模糊文字指令

特征:缺少上述四个要素中的任意一个,或要素之间打架(如"做个工具页但要营销感")。

**先分清楚是哪种模糊**——歧义类型决定处理路径:

#### 1.2.a 字段性歧义(缺细节)

特征:方向是清楚的("做个商品列表"),但具体字段、数量、是否带某个模块没说清。例:「做个商品列表」明确了页面类型,但没说"卡片是单列还是双列、要不要价格、要不要优惠标签"。

**处理路径**:

1. **写默认理解**——写一段「我理解这是 ___ 页面,给 ___ 用,主要展示 ___」交给伙伴看
2. **挑 1 个最关键的歧义反问**——不要一次抛 5 个问题让伙伴选,挑最影响后续生成的那 1 个
3. **得到澄清后**进 [`../AGENTS.md`](../AGENTS.md) §2 标准流程

#### 1.2.b 方向性歧义(缺主张)

特征:方向就是不清楚的——同一个需求可以做出几种**信息架构完全不同**的页面,反问 1 个字段救不回来。例:「做个橱窗模块」可以是「精选商品横向滚动」,也可以是「分类入口九宫格」,也可以是「主题活动 banner+商品行」,三个方向的优先级、决策逻辑都不同。

**处理路径**:

1. **先出 2-3 个信息架构方向**(不出图、只用文字描述)——每个方向说清楚:
   - 这个方向假设的用户主要焦虑是什么
   - 模块从上到下的优先级顺序
   - 用户在这一页要完成的核心动作
2. **伙伴选定一个方向后**再进 [`../AGENTS.md`](../AGENTS.md) §2 标准流程出真组件方案
3. **绝不在伙伴选定方向前直接出图**——选错方向再返工的成本远大于出文字方向的成本

**字段性 vs 方向性怎么分**:如果换一个对该页面没经验的同事来读需求,他能脑补出 2 种以上完全不同的页面长啥样 → 方向性;只能脑补出"细节不同但骨架一样"的版本 → 字段性。

#### 1.2 通用禁止

- 不读默认理解、不反问,**直接发挥审美**生成
- 把模糊当"自由发挥",往里塞营销 banner / 装饰元素以"显得丰富"
- 一次抛 5 个以上问题让伙伴回答(伙伴非技术背景,会被劝退)
- 在方向性歧义下**直接跳过出文字方向、用 1 个字段反问应付**——救不回来,伙伴答了你还是不知道哪个方向

## 2 · 低保真原型

特征:灰白线框、模块位置和优先级清楚,但**颜色/字号/间距/卡片样式都是占位的**——不准当真。

**核心原则**:低保真只管"放什么、先后顺序是什么、用户怎么走",不管"长什么样"。

**处理路径**:

1. **只取信息架构和优先级**——具体来说,提取这几类信息:
   - 页面目标(这是买/卖/租/营销/混合中的哪类页面)
   - 用户流程(用户从哪里来、要完成什么动作)
   - 模块顺序(页面有哪些模块,从上到下的优先级)
   - 信息优先级(哪些信息必须首屏出现)
   - 关键操作(哪些功能要组件承接)
2. **视觉表达完全回到设计系统**:颜色 / 字号 / 圆角 / 间距 / 卡片样式从 [`DESIGN.md`](./DESIGN.md) tokens 取,不继承低保真上的任何视觉值
3. 进 [`../AGENTS.md`](../AGENTS.md) §2 标准流程,按真组件填模块

**硬禁止(不能继承低保真的任何视觉值)**:

- 禁止继承低保真的**颜色**(线框灰、占位灰都不算视觉规范)
- 禁止继承低保真的**字号**(线框上的字号是占位)
- 禁止继承低保真的**间距 / 留白**
- 禁止继承低保真的**圆角 / 描边 / 阴影**
- 禁止把灰框占位变成真实 UI 风格的卡片样式(卡片用 `<ProductCardH>` / `<ProductCardFeeds>` 等真组件,不是按线框还原)
- 不要看到低保真没有的细节就跳过——空白区是"留白"或"按设计系统填",不是"原型没画就不做"

### 2.1 低保真图标污染(HARNESS-INPUT-V0.3)

低保真上的图标只用来理解"这个分类是什么",不能继承图标本身的视觉值——颜色、线框风格、图标密度、装饰方式都不算视觉规范。

**硬禁止**:

- 禁止把低保真上的图标颜色当成最终颜色照抄(线框稿里的蓝色/红色都是占位提示,不是品牌用色)
- 禁止为了"看起来不空"在工具型页面(设置 / 法务 / 表单 / 我的 / 隐私管理 / 订单设置等)批量补彩色分类图标——低保真没画图标的地方默认就不画
- 工具型页面如果设计系统没有明确对应的业务图标,优先用"纯文本列表 + chevron-right",不要自己挑图标名硬填

**必须用图标时的处理**:

- 默认走中性色 token:`var(--zd-color-text-tertiary)`(弱)或 `var(--zd-color-text-secondary)`(中等)
- 不允许整页或整组分类图标统一刷成 `var(--zd-color-link)` / `var(--zd-color-primary)` / 红色——这些 token 有专属语义,见 §2.2
- 删除/危险操作的分类图标也不要直接用红色 token 强调"危险"——危险表达走文案 + 按钮态,图标保持中性

### 2.2 token 语义边界(HARNESS-INPUT-V0.3,持续累积)

token 通过自查"有没有用 token"是不够的——还要看"token 的语义有没有用对"。这一节按真实反馈一条条累积,不一次列全。

**`var(--zd-color-link)`** — 蓝色 `#00A3FF`,链接 / 信息入口色

- ✓ 真链接("查看协议详情"、"前往隐私中心"等明确点击跳转的文字)
- ✓ 信息提示入口("为什么需要这些权限?"右侧的问号图标)
- ✓ 明确选中态的指示色(Tabs 当前项、Switch 开启)
- ✗ 工具型页面的分类装饰图标(设置项左侧、删除分组左侧、标签项左侧)
- ✗ 列表项右侧 chevron-right(用 `var(--zd-color-text-tertiary)`)
- ✗ 单纯为了"显得有重点"的图标着色

**`var(--zd-color-primary)` / 品牌红 `#FF0F27`** — 品牌强调色

- ✓ 品牌强调位(首页主 CTA、品牌 logo 区域)
- ✓ 危险操作按钮("确认删除"按钮)
- ✓ 关键 CTA(结算、提交)
- ✗ 普通图标装饰
- ✗ "看起来重要"就刷红——重要性走层级和文案,不靠红色

**`var(--zd-color-text)` / `-secondary` / `-tertiary` / `-muted`** — 中性前景色

- `--zd-color-text` `#111`:标题、主要文字
- `--zd-color-text-secondary` `#666`:副文案、说明文字
- `--zd-color-text-tertiary` `#999`:辅助文字、列表箭头、弱化图标的默认色
- `--zd-color-text-muted` `#BBB`:禁用态文字、极弱占位

判断不准时:工具型页面的图标和辅助元素,**默认走 `--zd-color-text-tertiary`,不走任何品牌色**。

### 2.3 图标风格默认(HARNESS-ICON-V0.1)

低保真和模糊文字指令场景下,业务装饰图标默认按 [`COMPONENT_USAGE.md`](./COMPONENT_USAGE.md) §"图标风格规则"的默认规则:**线稿、透明底、同页不混搭**。低保真上看到的图标颜色 / 底色块 / 面型表达都是占位提示,不照搬。

### 2.4 颜色意图三声明(HARNESS-COLOR-V0.2)

cursor / 业务 agent 写产物时,**默认所有文本/装饰颜色只能用以下白名单 token**:

- `var(--zd-color-text)` 主文(标题、列表左主、首要数值)
- `var(--zd-color-text-secondary)` 辅文(说明、列表右辅、次要时间戳)
- `var(--zd-color-text-tertiary)` 弱文(chevron、占位、辅助说明)
- `var(--zd-color-text-muted)` 极弱(禁用态文字)
- `var(--zd-color-primary)` 品牌红 `#FF0F27`(**低保真 + 语义重点**专用:异常值、危险操作、核心 CTA。不是"看起来重要就刷红")
- `var(--zd-color-primary-soft)` 品牌红淡底(与品牌红配对的选中浅底 / 优惠浅底)

**audit [9] 全字段硬拦**:扫所有着色字段(`color` / `background` / `background-color` / `border-color` / `border-top-color` / `border-right-color` / `border-bottom-color` / `border-left-color` / `outline-color` / `fill` / `stroke` / `iconColor` / `bgColor` / `textColor` / `themeColor` 等),任何未声明许可的 hex 直写(`#1682ff`、`#42a0ff` 等)、`rgb()` / `rgba()` / `hsl()`、或外部 `var(--*)` 一律 FAIL。中性兜底(`transparent` / `inherit` / `currentColor` / `#fff` / `#000` / `white` / `black`)放行。

**禁用 token**:`var(--zd-color-link)` 只能用在真链接、真导航位(`a` / `.link` / `.nav-link` 类),任何业务装饰类(`.row-value`、`.tag-num`、`.tag-mark`、`.item-value`)写蓝色 = audit / 自查 FAIL,反馈回流必抓。

**白名单不够用?三种放行声明,选一个**:

#### 模式 A:default(默认)

不需要任何声明。所有色严格走白名单,自查清单按白名单核。

#### 模式 B:user-spec(用户当条指令明说)

使用者在文字指令里明确点出某种颜色(比如"右值标红"、"用蓝色高亮"),且这种颜色不是设计系统标配。

**声明方式**(两件都要做):

1. 任一 jsx/tsx/html/css 文件首 5 行加注释:
   ```
   // HARNESS-COLOR-INTENT: user-spec
   ```
2. `.harness-evidence/USER-COLOR-SPEC.md` 摘录使用者原话,**推荐用 `user-quote:` 字段**:
   ```
   user-quote: "异常值要让人一眼看到,标红色"
   color: var(--zd-color-primary)
   scope: .row-value.is-anomaly
   ```
   兼容 markdown 引用块写法(`>`),audit 会 fallback 读引用块文字。

**audit [12] 颜色词硬校验(V0.2 新增)**:

audit 会扫 `user-quote:` 字段(或 markdown 引用块)里**有没有颜色词**——中文「红/蓝/绿/黄/紫/橙/黑/白/灰/青/粉/棕/褐/金/银」、英文 `red/blue/green/yellow/...`、hex/rgb/hsl 任一即可。

**没颜色词 → user-spec 不成立,硬 FAIL**。这是为了拦:

- 用户只说"参考图二样式",agent 不能据此声明 user-spec;
- 用户只说"用表单组件生成",agent 不能在 color 使用说明里编造"蓝色数字、蓝色图标";
- 颜色词**必须直接来自用户当条指令的原话**,不是 agent 自己的解读。

#### 模式 C:hifi-license(高保真稿带项目级颜色许可)

使用者交了高保真稿,且明确说"按这套颜色走"。这是项目级许可,本项目内复用,不是单条声明。

**声明方式**(三件都要做):

1. 任一 jsx/tsx/css 文件首 5 行加注释:
   ```
   // HARNESS-COLOR-INTENT: hifi-license
   ```
2. `.harness-evidence/PROJECT-COLOR-LICENSE.md` 登记许可:
   ```
   hifi-source: design-input/项目高保真.png
   approved-tokens:
     - var(--zd-color-link)  # 真链接位才用,装饰位还是禁
     - #2A8FFF               # 项目蓝(单点声明,不进 token)
   reason: 使用者明确"按高保真颜色走"
   ```
3. 高保真稿件本身放在产物里能找到(路径在 `hifi-source` 字段,自查时校验文件存在)。

audit / 自查看到声明 + 许可证据齐 + 稿件文件存在 → 放行该项目;缺一律 FAIL。

#### 默认 vs 例外的语义边界

| 场景 | 模式 | 说明 |
|------|------|------|
| 没附件、纯文字指令 | default | 全套白名单 |
| 低保真稿(线框/灰阶) | default | 低保真颜色不照搬 |
| 低保真稿 + "异常右值标红" | default | `--zd-color-primary` 在白名单内,不需切模式 |
| 低保真稿 + "用蓝色高亮" | user-spec | 蓝色不在白名单,必须 user-spec 声明 |
| 低保真稿 + "参考图二样式" | default | 用户没说颜色词,不算 user-spec |
| 高保真稿 | default | 默认仍走白名单(高保真色提取意图,不照搬) |
| 高保真稿 + 用户明说"按稿走色" | hifi-license | 项目级许可 |

**判断不准时**:默认走 `--zd-color-text-secondary`(辅文位)或 `--zd-color-text-tertiary`(弱文位),不切模式。切模式必须要么有用户原话要么有项目许可,否则一律 default。

### 2.5 跨产物参考规则(HARNESS-COLOR-V0.2)

**问题场景**:agent 拿历史 audit 红牌的旧产物当起点复制再改,把 `var(--zd-color-link)` 形式的蓝色换成 hex `#1682ff` 这种"换皮"操作绕过旧 audit 漏洞,污染就从一个产物传染到另一个产物。

**铁规**:

- **优先重新调真组件生成,不要拿别的产物当模板复制改**。
- 如果**确实**要拿别的产物当起点(同一项目的连续迭代、明确说"复用上一版"),必须在 `.harness-evidence/BASED-ON.md` 声明:
  ```
  source-project: ../privacy-personal-info-rights
  source-commit: 2f7ce3f7
  source-audit-status: pass
  reason: 在上一版隐私权益页面基础上加删除入口
  ```
- **`source-audit-status` 必须是 `pass`**——只能拿 audit 通过的产物当起点。red / warning 的产物 audit 一律 FAIL,不允许当模板。

**audit [13] 行为**:

- 有 `BASED-ON.md` → 校验三个字段齐 + `source-audit-status: pass`,否则 FAIL;
- 没有 `BASED-ON.md`,但扫到典型污染类名(`tag-mark` / `tag-num` / `row-value` / `item-value` 等历史污染产物常见的业务装饰类) → WARN 提示"是否复制了旧产物?建议声明 BASED-ON 或调真组件重生成"。

**为什么这条要硬卡**:低保真污染主要靠"产物间复制"传染,规则单纯卡颜色拦不住——agent 会换 hex 数值绕过。卡到来源溯源,让"参考即留痕",这条传染链才断得掉。

## 3 · 高保真原型

特征:已有完整视觉表达,颜色/字号/间距/卡片都看着像最终交付。

**核心原则**:高保真用来理解视觉意图,**不照搬具体样式数值**。所有高保真都按同一规则处理,不管它来源是当前设计系统、竞品、还是临时活动稿——agent 没法可靠判断图片来源,安全做法是不区分。

**处理路径**:

1. **提取视觉意图**——从高保真上读出这些信息:
   - 层级(什么模块在主视觉位、什么是次要的)
   - 状态表达(默认 / 选中 / 禁用 / loading / 空态等)
   - 信息密度(一屏放多少模块、卡片多紧凑)
   - 视觉权重(哪些信息要强调、哪些要弱化)
   - 情绪强度(工具感 / 营销感 / 节日感)
2. **不照搬具体样式数值**——颜色 hex、字号 px、间距 px、圆角、阴影都不直接抄,最终颜色/字体/间距/组件形态全部回到设计系统的 tokens 和组件
3. **开工前简短说明会用到哪些组件**——在回复里用一句话列出,例如"会用到 `<TopNav>` / `<ProductCardH>` / `<Price>`,原图右上角的 banner 区没有现成组件,会标 custom module"。不要求列表格,一行话即可
4. **允许在设计系统范围内做锦上添花**——层级优化、留白优化、样式 polish、状态补全都可以做,前提是颜色/字体/间距全部走 tokens、组件用真组件
5. 进 [`../AGENTS.md`](../AGENTS.md) §2 标准流程,按反查结果调真组件 + 用 tokens 写 custom module

**硬禁止**:

- **直接照搬高保真的 CSS 数值**(颜色 hex、字号 px、阴影、圆角的具体数值)——高保真只是参照,最终视觉走 tokens
- **看到一个已有组件能覆盖的模块,仍然手写一个"贴近高保真"的版本**(这违反 [`../AGENTS.md`](../AGENTS.md) §1 铁律一)
- **把高保真上的"装饰元素"(渐变背景、毛玻璃、阴影)当默认要求**——工具型页面这些是禁的([`DESIGN.md`](./DESIGN.md) §7),除非这是营销页面(→ §5)
- **以"高保真来自我们设计系统"为由直接照搬样式**——agent 没法可靠判断来源,一律按"提取意图、回到 tokens"处理

**冲突处理**:高保真上的视觉值和设计系统 tokens 不一致时(颜色差几度、间距差 4px 等),**以设计系统为准**,除非 PRD 显式说明"这是营销例外"。

**图标风格例外(HARNESS-ICON-V0.1)**:

只有当**输入源是高保真稿** AND **使用者在文字指令里明确点出"参考此图标风格"或"图标风格按稿走"**两条同时成立时,才允许在生成产物里出现:

- 装饰图标使用面型 / `solid` 风格
- 装饰图标外裹底色(圆形 / 圆角矩形等色块)

只满足其中一条不成立,仍按 [`COMPONENT_USAGE.md`](./COMPONENT_USAGE.md) 默认规则——线稿、透明底、同页不混搭。例外触发时,agent 在 `.harness-evidence/` 写一行说明:"image=<高保真文件名>; instruction=<使用者原话>",自查/反馈回流凭此条放行。无证据 = 默认规则 = 自查报错。

## 4 · PRD / 业务上下文

特征:文字 + 表格 + 业务规则,描述"做什么、为什么做、给谁用、关键流程是什么"。

**处理路径**:

1. **先提炼 UX Brief**——别直接根据 PRD 开工。UX Brief 包含五项:
   - **页面目标**:用户在这一页能完成的核心动作
   - **目标角色 + 模式**:买家 / 卖家 / 租赁 / 回收,C 端 / 商家端
   - **核心信息**:必须出现的字段(标题、价格、CTA 等),不必出现的("产品功能介绍"、"品牌故事"等通常不在 C 端工具页)
   - **风险点**:营销场景判定 / 字段语义边界(如价格是否带后缀、价格区间)/ 信息密度
   - **推荐组件**:从 [`COMPONENT_USAGE.md`](./COMPONENT_USAGE.md) 反查
2. **UX Brief 写成 DECISIONS.md 一段**(业务项目根目录),让伙伴看一眼确认再开工
3. 进 [`../AGENTS.md`](../AGENTS.md) §2 标准流程

**禁止**:

- **不出 UX Brief 直接开始写代码**——PRD 信息密度高,跳过提炼会丢核心字段
- **把 PRD 里"产品功能介绍"等市场材料当页面内容塞**——C 端工具页只放工具内容
- 把 PRD 里出现的"运营希望强营销感"当 §5 营销场景的判定依据——必须命中 [`DESIGN.md`](./DESIGN.md) §8.1 三条任一才走 §5

## 5 · 营销 / 大促 / 节日活动

特征:命中 [`DESIGN.md`](./DESIGN.md) §8.1 三条任一即视为营销场景。

**处理路径**:

1. **先判断"营销主页面" vs "普通页面里的活动入口"**:
   - 营销主页面 = 整页就是活动会场 / 节点专题 / 抽奖落地 / 大促集合
   - 活动入口 = 工具页(首页 / 商详 / 列表)里嵌一块活动卡 / 倒计时条 / 入口模块
2. **营销主页面**:用 [`DESIGN.md`](./DESIGN.md) §8 营销例外条款 + 在业务项目根目录的 DECISIONS.md 显式声明"本页面属营销场景,已套用 §8 例外"。视觉张力可以强,但**必须明确是"整页营销"而不是"工具页加营销外壳"**
3. **活动入口**:用现有组件 + 局部样例。营销张力**只在该模块内**,不能渗透到同页商品卡 / 列表项 / 商详正文([`DESIGN.md`](./DESIGN.md) §8.3)
4. 在 DECISIONS.md 标注"本模块属营销场景,已套用 [`DESIGN.md`](./DESIGN.md) §8 例外条款"

**禁止**:

- **不命中 §8.1 三条却套营销例外**——伙伴一句"做得好看点 / 抢眼点"不是营销判定依据
- **把营销视觉渗透到工具型组件**(商品卡、商详、订单、表单等)—— [`DESIGN.md`](./DESIGN.md) §8.3 明确禁
- **使用 emoji**——营销场景仍禁([`DESIGN.md`](./DESIGN.md) §8.2)
- **使用绝对化文案**:"全网最低 / 独家 / 史上最便宜 / 错过等一年" 仍禁([`DESIGN.md`](./DESIGN.md) §8.2)

## 6 · 混合输入

伙伴常常一次给:PRD + 低保真,或 PRD + 高保真,或 PRD + 文字补充 + 营销描述。

**处理路径(按角色定权重)**:

| 输入源 | 角色 | 权重 |
|-|-|-|
| PRD | 定业务目标、核心字段、流程 | **业务正确性最高** |
| 低保真 | 定信息架构、模块顺序、优先级 | 结构层 |
| 高保真 | 给视觉意图(不是最终值) | 参照层 |
| 设计系统(tokens / 组件 / DESIGN.md) | 定最终视觉表达 | **视觉正确性最高** |

**冲突时的判定**:

- **业务字段 vs 视觉**:PRD 说要展示价格区间,但 `<Price>` 组件不支持后缀 → 找专用组件 / 起 ListRow 兜底,不在 `<Price value>` 里硬塞后缀(参考 [`COMPONENT_USAGE.md`](./COMPONENT_USAGE.md) HARNESS-001)
- **低保真布局 vs 设计系统组件**:低保真画的卡片样式和 `<ProductCardH>` 不一样 → 用 `<ProductCardH>`,低保真当结构参考
- **高保真颜色 vs tokens**:高保真用了 `#FF1234`,tokens 没有这个值 → 用 tokens 里最近的品牌红(`var(--zd-color-primary)` `#FF0F27`),除非是营销场景
- **多个输入源都说"营销"但都没命中 §8.1**:仍按工具型处理,反问伙伴"是否走营销例外"

**禁止**:

- **不分权重直接照搬最容易看到的那份输入**(如只看高保真不看 PRD,或只看 PRD 不看低保真)
- **遇到冲突就找折中方案**(如 `<Price value>` 里塞"约"字混过去)——按上面权重判,按规则做
- **把所有输入源叠在一起当"全部要求"**,导致页面信息密度爆炸——核心信息按 PRD,结构按低保真,视觉按设计系统

## 7 · 兜底:无现成组件 / 设计系统没覆盖

任何输入源走到 [`../AGENTS.md`](../AGENTS.md) §2 第 3 步(按规范泛化)都可能遇到:设计系统里**确实没有对应组件**。

**处理路径**:

1. **先在 [`COMPONENT_USAGE.md`](./COMPONENT_USAGE.md) 组件清单里反查**——18 个聚合 barrel 里有没有相似命名 / 相似职责的组件被你忽略了
2. **再用 tokens + 基础原子组合 custom module**——颜色 / 字体 / 圆角 / 间距全部从 [`DESIGN.md`](./DESIGN.md) tokens 取,营销场景按 §8 例外
3. **声明 custom module 清单**——在业务项目根目录的 DECISIONS.md(如果没有就建一个)里列出:
   - custom module 名称
   - 对应的设计系统组件为什么不能用("找过 X、Y、Z 都不匹配")
   - 用了哪些 tokens
   - 是否走了 §8 营销例外
4. **不要写回设计系统**——custom module 只活在业务项目里([`../AGENTS.md`](../AGENTS.md) §1 铁律一)。要回写设计系统是维护者通过 PR 做的事,不是 agent 的自由。

**禁止**:

- **伪造设计系统组件**——给 custom module 起一个像 `<ProductCardXxx>` 的"组件名"假装是 `src/components/Product.tsx` 导出的,是禁止的
- **把"反复出现的 custom module"自己回写进 `src/components/`**——那是维护者的 PR 流程,不是 agent 行为
- **跳过组件清单查询**直接造新模块——18 聚合 barrel 里很多场景已经有了

## 8 · 这份文档没覆盖的,怎么办

V0 只写六类输入源 + 兜底,没覆盖的输入形态(如截图、Figma 链接、口头转录等):

- 先按上面六类里**最接近的一类**走
- 把"我把它当哪一类处理了 + 为什么"写进 DECISIONS.md
- 反馈给主仓维护者(按 [`HARNESS_FEEDBACK.md`](./HARNESS_FEEDBACK.md)),下一版迭代会扩

V0 的边界就是「先把六类常见输入源对上」。后续迭代靠反馈回流([`CHANGELOG-RULES.md`](./CHANGELOG-RULES.md) + 反馈通道),不是一次写满。

## 关联 Harness 规则

- `HARNESS-INPUT-V0`:六类输入源处理路径 + 兜底(cold-start 来源,不是真实反馈数据)
- `HARNESS-INPUT-V0.1`:图片附件不是默认视觉稿,agent 必须先看图判断角色;低保真硬禁止继承视觉值;高保真不照搬数值但允许设计系统范围内 polish;伙伴说法和图片冲突优先按图片,不确定才兜底低保真
- `HARNESS-INPUT-V0.2`:复杂图片输入(≥3 张/多屏拼接/流程箭头/≥3 状态/明说严格还原/≥4 模块)必须先输出附件理解摘要再进入代码生成;伙伴一回合无异议自动继续
- `HARNESS-INPUT-V0.3`:低保真图标颜色不能继承(§2.1),工具型页面默认不补彩色分类图标;token 语义边界开始按反馈累积(§2.2),首批沉淀 `--zd-color-link` / `--zd-color-primary` / `--zd-color-text/-secondary/-tertiary/-muted` 用法
- `HARNESS-COLOR-V0.2`:颜色意图三声明(§2.4)——default / user-spec / hifi-license,白名单不够用必须显式声明 + 证据齐;user-spec 必须有用户原话颜色词;audit [9] 全字段硬拦非白名单颜色
- `HARNESS-COLOR-V0.2`:跨产物参考规则(§2.5)——复用旧产物必须声明 `BASED-ON.md`,且来源审查状态必须为 pass
- `HARNESS-ICON-V0.1`:图标风格默认(§2.3)与高保真例外(§3 末尾)

具体规则演进史见 [`CHANGELOG-RULES.md`](./CHANGELOG-RULES.md)。
